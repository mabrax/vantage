# Structure Outline: Pinned Deno and Codex app-server compatibility spike

Build the spike as six independently verifiable increments, beginning with the immutable version/schema contract and ending with atomic publication of a real authenticated run. Deterministic fake-child tests establish each transport, lifecycle, diagnostic, and shutdown property before credentials are involved; the live acceptance path remains fail-closed unless macOS arm64 containment proves that even an immediately escaped descendant is tracked and terminated.

## Desired End State

- Deno `2.9.3`, Codex CLI `0.145.0`, stable generator mode, macOS arm64 validation scope, safety limits, and deterministic generated-bundle hashing are recorded in one immutable compatibility manifest.
- The unmodified TypeScript and draft-07 JSON Schema artifacts generated by the pinned Codex CLI are committed with generation metadata and a complete stable-surface coverage manifest.
- A Deno harness launches `codex app-server` directly, drains stderr separately, preserves bounded JSONL wire order, correlates interleaved responses, and rejects malformed, unknown, oversized, or out-of-order protocol behavior.
- Exact-version, missing-executable, unauthenticated-account, artifact-mismatch, and unsupported-platform states fail before an authenticated turn with stable actionable diagnostics.
- One real authenticated turn in a temporary Git repository completes `initialize`, `initialized`, `account/read`, paginated `model/list`, `thread/start`, `turn/start`, streamed item output, and terminal `turn/completed`.
- The retained bidirectional journal and redacted transcript prove native thread/turn/item continuity, schema validity before and after redaction, required lifecycle ordering, and a non-empty completed agent message without retaining credentials or direct personal data.
- Shutdown is memoized and bounded through stdin close, process-group `SIGTERM`, and process-group `SIGKILL`; direct-child status, drains, observed descendants, signal path, and timings are retained.
- `noObservedDescendantsRemain` and `escapedDescendantContainmentProven` remain separate facts. The absolute cleanup gate is false unless race-closing containment or event tracking passes an immediate-`setsid` fake-child test and the real Codex run.
- Acceptance publishes the run-derived coverage manifest, transcript, and summary as one cross-hashed proof set; stale or partially replaced evidence never validates the pair.

## Validation Environment

- **Target**: Repository-owned Deno `2.9.3` tasks. Deterministic checks use fake children over real stdio without credentials or network access; compatibility acceptance uses Codex CLI `0.145.0` with stable app-server schemas on the manifest-selected `darwin`/`aarch64` target.
- **Preflight checks**: Confirm the repository is at the intended revision; confirm `deno --version`, `codex --version`, and `git --version` are available with exact supported versions; run `deno task protocol:verify`; for live tasks, confirm the selected `CODEX_HOME` already returns a non-null account and the current platform matches the manifest.
- **Migration/setup policy**: No database or service migration is allowed. Generated protocol trees may be regenerated only by the exact pinned CLI in stable mode, through `deno task protocol:generate`, into temporary output before deterministic comparison or replacement. Tests may create disposable Git repositories and temporary evidence directories; they must not create, copy, or modify credentials.
- **Blocked means**: Deterministic work is blocked only by an unavailable exact Deno toolchain or an unreproducible pinned schema bundle. A trustworthy compatibility claim is blocked by missing/mismatched Codex, absent authentication, a non-macOS-arm64 target, a live protocol failure, stale hashes, remaining processes, or any failure to prove escaped-descendant containment. Snapshot polling alone is explicitly insufficient for the absolute cleanup gate.

## UX Readiness

**Level**: foundation-only

This spike creates runtime compatibility evidence and actionable CLI diagnostics only. It does not create a window, WebView, route, screen, browser flow, product copy, or presentable conversation UX.

- [ ] Phase 1: Establish the immutable compatibility and generated-protocol contract

- [ ] Phase 4: Prove bounded shutdown and escaped-descendant containment fail-closed

## Phase 1: Establish the immutable compatibility and generated-protocol contract

Introduce the smallest executable repository foundation that can pin the runtime pair, regenerate the stable Codex contract, compile its schemas correctly, and prove that every generated stable method is classified exactly once. This phase ends with reproducible generated inputs and offline configuration/schema tests; it does not launch app-server or claim the pair is validated.

### Dependencies and Parallelism

- **Depends on**: None
- **Can run in parallel with**: None
- **Parallel ownership boundary**: Not applicable; this phase defines the task names, dependency pins, manifest shapes, generated paths, and shared protocol types consumed by every later phase.

### File Changes

- **`deno.json`**: Pin non-generated dependencies, including the draft-07 Ajv boundary, and define `check`, `protocol:generate`, `protocol:verify`, `spike:verify`, and `spike:accept` tasks with the permissions described by the TDD.
- **`spikes/codex-app-server/compatibility.json`**: Record immutable Deno/Codex versions, stable generator commands, generated root and hash, macOS arm64 validation policy, JSONL/queue/page bounds, and lifecycle/shutdown safety timeouts. Do not store mutable candidate/validated status or a run-specific coverage hash.
- **`spikes/codex-app-server/generated/0.145.0/generation.json`**: Record exact commands, `experimental: false`, actual generated file counts, generation time, and the deterministic bundle hash.
- **`spikes/codex-app-server/generated/0.145.0/types/**`**: Commit the unmodified stable TypeScript output from Codex CLI `0.145.0`.
- **`spikes/codex-app-server/generated/0.145.0/json-schema/**`**: Commit the unmodified stable JSON Schema output from Codex CLI `0.145.0`.
- **`spikes/codex-app-server/coverage.json`**: Add a complete initial classification derived from generated union membership, with zero observations until an explicit acceptance run derives replacements.
- **`spikes/codex-app-server/schemas/compatibility.schema.json`**: Validate immutable manifest structure, exact literal pins, positive bounds, and supported validation-platform fields.
- **`spikes/codex-app-server/schemas/coverage.schema.json`**: Validate coverage directions, dispositions, rationales, counts, bundle binding, and uniqueness.
- **`spikes/codex-app-server/src/config.ts`**: Load and validate the manifest and generation metadata, normalize repository-relative paths, calculate canonical file/tree hashes, and distinguish current immutable inputs from any derived validation status.
- **`spikes/codex-app-server/src/generate_protocol.ts`**: Assert exact Codex version, generate both trees into a temporary sibling, hash path-sorted `path + NUL + bytes + NUL` records, compare or replace atomically, and never edit generated files.
- **`spikes/codex-app-server/src/protocol_validation.ts`**: Compile every generated top-level draft-07 schema with range-validating `uint`, `uint16`, `uint32`, and `int64` formats; leave unknown formats fatal and expose method-specific request/response/notification validation.
- **`spikes/codex-app-server/src/coverage.ts`**: Extract all stable client-request, client-notification, server-notification, and server-request methods from generated unions and require one coverage entry per generated method.
- **`spikes/codex-app-server/tests/config_test.ts`**: Cover manifest/schema rejection, immutable hash binding, path normalization, and generated-bundle mismatch.
- **`spikes/codex-app-server/tests/protocol_validation_test.ts`**: Compile every generated top-level schema under the configured draft-07 validator and cover Codex numeric-format boundaries and unknown formats.
- **`spikes/codex-app-server/tests/coverage_test.ts`**: Prove generated membership and coverage membership are equal sets with no duplicates, omissions, or experimental methods.

### Validation

#### Automated Verification

- [ ] `deno task protocol:generate`
- [ ] `deno task protocol:verify`
- [ ] `deno test spikes/codex-app-server/tests/config_test.ts spikes/codex-app-server/tests/protocol_validation_test.ts spikes/codex-app-server/tests/coverage_test.ts`
- [ ] Regenerate into a fresh temporary directory and verify the file counts and deterministic bundle hash match the committed metadata without modifying the committed generated tree.

#### Runtime Verification

None; generated-contract reproducibility and schema compilation are fully automated in this phase.

## Phase 2: Prove bounded bidirectional JSONL transport over a real child process

Add a real subprocess seam and a scripted fake app-server so framing, serialized writes, response correlation, observation order, schema dispatch, queue bounds, EOF behavior, and stderr isolation are verified end to end over actual pipes. The slice ends at ordered validated protocol records; it does not yet interpret the conversation lifecycle or publish evidence.

### Dependencies and Parallelism

- **Depends on**: Phase 1
- **Can run in parallel with**: None
- **Parallel ownership boundary**: Not applicable; this phase establishes the `ProcessHost`, fake-child, `JsonlClient`, validator-dispatch, and record interfaces that Phases 3 and 4 consume.

### File Changes

- **`spikes/codex-app-server/src/diagnostics.ts`**: Define stable typed diagnostics for process, framing, schema, correlation, queue, timeout, and close failures, with stage and actionable context fields.
- **`spikes/codex-app-server/src/process_host.ts`**: Launch an executable plus argument array with piped stdin/stdout/stderr, expose direct status, drain bounded stderr concurrently, and delegate final termination to the shutdown boundary without parsing stderr as protocol.
- **`spikes/codex-app-server/src/jsonl_client.ts`**: Implement one serialized writer, register-before-write request correlation with rollback, bounded byte framing, strict UTF-8/JSON parsing, complete-frame indexing before validation, response dispatch, and an ordered bounded message queue.
- **`spikes/codex-app-server/src/transcript.ts`**: Add the in-memory bidirectional record primitive: client records immediately before exact writes, server records for every complete stdout frame, shared `observationIndex`, server-only `wireIndex`, monotonic offsets, and raw diagnostic metadata for invalid frames.
- **`spikes/codex-app-server/tests/fixtures/fake_app_server.ts`**: Provide a real subprocess fixture that can split/coalesce frames, interleave responses and notifications, emit malformed/unknown/oversized input, delay reads, close early, leave requests pending, write large stderr output, ignore signals, and select descendant behaviors needed by Phase 4.
- **`spikes/codex-app-server/tests/jsonl_client_test.ts`**: Cover partial and multiple frames, concurrent response correlation, register-before-write races, serialized writes, duplicate/unknown IDs, second initialization, post-close writes, invalid UTF-8/JSON, oversized lines, incomplete EOF, pending-request EOF, queue count/byte exact boundaries and overflow, stdout order, and independent stderr drainage.
- **`spikes/codex-app-server/tests/protocol_validation_test.ts`**: Extend generated-schema dispatch tests to exercised client and server envelopes and prove malformed or unknown methods are recorded before the exact-version run fails.

### Validation

#### Automated Verification

- [ ] `deno test spikes/codex-app-server/tests/jsonl_client_test.ts`
- [ ] `deno test spikes/codex-app-server/tests/protocol_validation_test.ts`
- [ ] `deno task protocol:verify`
- [ ] Verify exact-boundary queue cases succeed, first-byte-over-bound cases fail without dropping or reordering a complete frame, and stderr pressure cannot block stdout processing.

#### Runtime Verification

None; the fake app-server runs as a real subprocess, so the transport behavior is covered by automated integration tests.

## Phase 3: Prove preflight, lifecycle, transcript, and coverage behavior offline

Compose the transport into the required compatibility scenario using the fake app-server: exact environment checks, initialization, authentication preflight, bounded model pagination, native thread/turn/item state, raw-before-redaction validation, schema-preserving retained evidence, and journal-derived coverage. This phase proves the complete scenario policy without credentials, network variability, committed evidence writes, or an absolute shutdown claim.

### Dependencies and Parallelism

- **Depends on**: Phase 1 and Phase 2
- **Can run in parallel with**: Phase 4
- **Parallel ownership boundary**: This phase owns `preflight.ts`, `lifecycle_scenario.ts`, lifecycle/transcript policy, preflight/transcript tests, and local evidence schemas. Phase 4 owns shutdown, process-tree tracking, and shutdown tests; both consume the Phase 2 host/fixture contracts without changing their public symbols.

### File Changes

- **`scripts/run-protocol-spike`**: Diagnose a missing developer Deno as `DENO_NOT_FOUND`, report the exact expected version and recovery action, and hand off to the Deno task without constructing a Codex command.
- **`spikes/codex-app-server/schemas/evidence.schema.json`**: Validate retained client/server records, observation/wire indexes, schema proof, native ID indexes, measurements, shutdown facts, cross-hashes, and acceptance gates.
- **`spikes/codex-app-server/src/preflight.ts`**: Enforce exact Deno/Codex versions, executable resolution, artifact hash, validation platform in acceptance mode, and `account !== null` authentication; do not reject a non-null account solely because `requiresOpenaiAuth` is true.
- **`spikes/codex-app-server/src/lifecycle_scenario.ts`**: Create a disposable Git repository; perform initialize/initialized/account/model/thread/turn flow; terminate model pagination on null or absent cursor; reject repeated cursors and page/deadline exhaustion before `thread/start`; and drive the native lifecycle reducer to a terminal turn.
- **`spikes/codex-app-server/src/transcript.ts`**: Add semantic lifecycle validation, native identity continuity, item start/delta/completion rules, agent-text reconstruction, non-empty completed-agent-message enforcement, deterministic same-type redaction, post-redaction pinned-schema validation, and local evidence-schema validation.
- **`spikes/codex-app-server/src/coverage.ts`**: Derive observed counts and dispositions from both outbound and inbound schema-valid journal records, retain optional-event semantics, and fail on disagreement without mutating committed coverage in verify-only mode.
- **`spikes/codex-app-server/tests/preflight_test.ts`**: Cover missing Deno, Deno mismatch, missing Codex/`NotFound`, Codex mismatch, artifact mismatch, unsupported platform, non-null authenticated account, null account rejection before thread start, null/absent/repeated model cursors, page bound, and catalog deadline.
- **`spikes/codex-app-server/tests/transcript_test.ts`**: Cover request/thread/turn/item identity errors, invalid lifecycle transitions, delta reconstruction, terminal statuses, raw-before-redaction behavior, sensitive-field rejection, schema-preserving replacements, observation/wire ordering, and outbound/inbound coverage derivation.

### Validation

#### Automated Verification

- [ ] `deno test spikes/codex-app-server/tests/preflight_test.ts`
- [ ] `deno test spikes/codex-app-server/tests/transcript_test.ts`
- [ ] Run the scripted full-lifecycle fake scenario and assert no request after a failing preflight, exactly one initialization, bounded model pagination, one completed native turn, a non-empty agent message, and a schema-valid retained journal.
- [ ] Run `deno task protocol:verify` against both zero-observation baseline coverage and a temporary journal-derived coverage result.

#### Runtime Verification

None; the phase deliberately uses scripted protocol behavior so every lifecycle and diagnostic outcome is deterministic.

## Phase 4: Prove bounded shutdown and escaped-descendant containment fail-closed

Implement one idempotent shutdown state machine for every success and failure exit, with macOS arm64 process-group signaling and explicit lineage evidence. Keep snapshot observations separate from the absolute gate: this phase may complete only when a race-closing containment or event-tracking implementation continuously covers child creation through shutdown, passes the immediate-`setsid` escape fixture, and can truthfully set `escapedDescendantContainmentProven`; otherwise the phase and every compatibility-validation claim remain blocked.

### Dependencies and Parallelism

- **Depends on**: Phase 1 and Phase 2
- **Can run in parallel with**: Phase 3
- **Parallel ownership boundary**: This phase exclusively owns `shutdown.ts`, `process_tree_darwin.ts`, shutdown-specific fixture modes, and `shutdown_test.ts`. It integrates with the Phase 2 host through the fixed close/status/stream interfaces and does not change lifecycle or transcript policy owned by Phase 3.

### File Changes

- **`spikes/codex-app-server/src/shutdown.ts`**: Memoize `close()`, reject new work, settle pending requests, conditionally request turn interruption, close stdin, await bounded direct status, escalate to a verified positive-root process group with `SIGTERM` then `SIGKILL`, await drains/status, and return structured shutdown evidence on every exit path.
- **`spikes/codex-app-server/src/process_tree_darwin.ts`**: Implement the macOS arm64 process-group and transitive-lineage boundary. Expose `noObservedDescendantsRemain` from post-shutdown inspection separately from `escapedDescendantContainmentProven`; the latter may become true only when the tracker has race-closing, overflow-free coverage from spawn through final verification and accounts for descendants that create a new session or are reparented.
- **`spikes/codex-app-server/src/process_host.ts`**: Start the child in the owned session/process group, initialize containment tracking before child work can escape its observation boundary, guard negative-PID signaling with root/session identity checks, and route success, protocol failure, timeout, cancellation, and unexpected exit through the same close promise.
- **`spikes/codex-app-server/src/diagnostics.ts`**: Add stable `DESCENDANT_LEAK`, containment-unproven, shutdown-timeout, and unsafe-process-group diagnostics with observed PIDs and shutdown stage but without overstating snapshot evidence.
- **`spikes/codex-app-server/tests/fixtures/fake_app_server.ts`**: Add deterministic modes for graceful exit, ignored stdin close, ignored `SIGTERM`, ordinary grandchildren, grandchildren surviving parent exit, and an immediate `setsid`/reparent escape.
- **`spikes/codex-app-server/tests/shutdown_test.ts`**: Cover concurrent close calls, every close entry path, graceful/TERM/KILL escalation, direct status and drain completion, root/session guards, ordinary descendant cleanup, remaining-PID failure, and the escaped-descendant absolute gate.

### Validation

#### Automated Verification

- [ ] `deno test spikes/codex-app-server/tests/shutdown_test.ts`
- [ ] Verify the graceful, `SIGTERM`, and `SIGKILL` fixtures each terminate within manifest bounds and leave no direct child or observed descendant.
- [ ] Verify snapshot polling alone produces `noObservedDescendantsRemain: true` at most and leaves `escapedDescendantContainmentProven: false`.
- [ ] Verify the immediate-`setsid` fixture is tracked and terminated by the race-closing mechanism before the absolute gate may become true; if it escapes tracking or cleanup, assert the phase fails with the pair still a candidate.
- [ ] Run the protocol-failure, timeout, unexpected-request, child-exit, and cancellation fixtures and assert they all share the same memoized shutdown path.

#### Runtime Verification

None; the required process behaviors, including the adversarial escaped-descendant case, must be repeatable automated tests rather than exploratory process-table inspection.

## Evidence Gaps

- The exact stable generated file tree, actual file counts, and deterministic bundle hash are not known until Codex CLI `0.145.0` performs stable generation. Researched counts are validation evidence, not values to hard-code.
- Available repository, Deno, and TDD evidence does not name a proven macOS arm64 primitive that closes the fork/`setsid`/reparent observation race. Phase 4 must either supply a continuously covering containment or event-tracking implementation that passes the immediate-escape fixture, or remain blocked with `escapedDescendantContainmentProven: false`; process-group signaling and snapshot polling alone cannot satisfy the ticket's absolute descendant gate.
- The selected Deno/Codex pair remains a candidate until Phases 5 and 6 complete on the declared platform with an existing authenticated `CODEX_HOME` and every cross-hashed gate true.
- Startup, first-event, completion, and shutdown measurements remain unknown until the live run. The implementation records them and applies bounded safety timeouts but must not invent performance acceptance thresholds.
- Optional live notifications cannot be predicted. Their presence or absence is derived from the accepted journal and never changes the required lifecycle.
